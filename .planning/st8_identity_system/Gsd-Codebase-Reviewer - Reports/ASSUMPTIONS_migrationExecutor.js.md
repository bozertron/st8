The report generator — which IS connected — tells users to run `integr8 --apply`. This command would need to:
1. Load the migration plan from `migration_plan.toml`
2. Execute it step by step
3. Verify the result

All three capabilities exist in `migrationExecutor.js`:
- `loadMigrationPlan()` — loads TOML plan
- `executeMigrationPlan()` / `executeAtomicMigration()` — executes steps
- `verifyIntegration()` — 4-level verification pipeline

**But no `--apply` handler exists anywhere.** This is a documented feature gap, not dead code.

### Evidence B: Data Flow Design Is Clear

The integr8 pipeline has a clear 3-stage design in `index.js`:
```
Stage 1: Data Ingestion (dataIngestion.js) → builds SemanticGraph
Stage 2: Relationship Analysis (relationshipAnalyzer.js) → finds conflicts
Stage 3: Path Generation (pathGenerator.js) → generates MigrationPlan
Stage 4: [MISSING] Execute Plan → migrationExecutor.js (NEVER CALLED)
```

`index.js` generates a `MigrationPlan` with typed steps (`COPY_FILE`, `REWRITE_IMPORT`, `MERGE_ROUTE`, `RESOLVE_CONFLICT`, `RUN_COMMAND`, `VERIFY`), serializes it to `migration_plan.toml`, and writes it to disk. Then... nothing. The plan is generated but never executed.

## Recommendation: **REWIRE**

### Why NOT Delete:
1. The report generator explicitly references `--apply` and `--verify` commands that need this module
2. The type system (`MigrationAction` enum) is shared across pathGenerator, reportGenerator, and migrationExecutor — this was intentional design
3. The TOML serialization/deserialization pair was built for the generate→execute pipeline
4. 1,837 lines of working, tested-capable code with 11 public APIs, rollback support, verification pipeline, and atomic migration
5. All internal dependencies (`types.js`, `tomlSerializer.js`) are satisfied

### What Needs to Happen:
1. **Primary:** Add `migrationExecutor` import to `index.js` and wire `--apply` / `--verify` CLI flags
2. **Secondary:** Consider making `executeAtomicMigration` the default execution mode (with snapshot + rollback)
3. **Tertiary:** Add `RUN_COMMAND` step generation to `pathGenerator.js` for post-migration commands (npm install, build, etc.)
4. **Security:** Review `executeSandboxedTransform()` before production — it uses `new Function()` for user-defined transforms
5. **Cleanup:** Remove the phantom `./views/NewView.vue` default from `executeMergeRoute()` line 362

### Where to Wire It:

**Option A — Extend `index.js`:**
```js
// Add to index.js imports:

```

**Option B — Separate CLI command:**
Create a new CLI entry point `integr8-apply` or add subcommands to the existing CLI that maps `--apply` to `migrationExecutor.js`.
