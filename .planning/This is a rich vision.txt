## Refined Proposal: The File Fingerprint System v2

### 1. The Fingerprint Is Not a Hash — It's an Identity

You're right to push back. A content hash that breaks on every edit is a **version stamp**, not a fingerprint. What you're describing is a **file identity** that's stable from concept through production, with content versions as a time-series attached to it.

Here's the architecture:

**Fingerprint = composite key: `{filepath}:{birthTimestamp}`**

- `filepath` — survives renames if we track them (move = same identity, new path)
- `birthTimestamp` — set once at file creation (first `add` event from file watcher), never changes
- `sha256Hash` — demoted from PK to a **content version field**, updated on every change
- Every time the hash changes, a **mutation record** is appended to a `file_mutation_log` table

This gives you the full "path to production" you described:

| Phase | What's Recorded |
|-------|----------------|
| **Concept** (pre-code) | File doesn't exist yet. Fingerprint registered in DB with `status: CONCEPT`, intent metadata, roadmap references, assumed types |
| **MVP Lock** | All concept files emit schema cards simultaneously. Cross-file validation runs. PRD falls out. |
| **Wiring** | Dependencies validated against schema cards. Type mismatches flagged before code exists. |
| **Development** | Every file change → mutation record logged (who, what, when, diff hash). File memory grows. |
| **Production** | Mutation log purged (or archived). Lightweight schema card replaces full development memory. |

### 2. The Persistence Mechanism: Not .md Files — SQLite + Hook-Driven Emission

Your instinct about .md files drifting is 100% correct. Here's what actually works:

**Primary storage: `file_mutation_log` table in SQLite**

```sql
CREATE TABLE IF NOT EXISTS file_mutation_log (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  fingerprint TEXT NOT NULL,          -- stable identity
  sha256_hash TEXT NOT NULL,          -- content version at this mutation
  mutation_type TEXT,                 -- 'CONCEPT' | 'CREATE' | 'EDIT' | 'RENAME' | 'REFACTOR' | 'LOCK' | 'PRODUCTION'
  changed_fields TEXT,                -- JSON diff of what changed
  actor TEXT DEFAULT 'DEVELOPER',     -- 'DEVELOPER' | 'INDEXER' | 'WATCHER' | 'AGENT'
  timestamp TEXT DEFAULT CURRENT_TIMESTAMP,
  metadata TEXT,                      -- JSON blob for schema card, type signatures, etc.
  FOREIGN KEY (fingerprint) REFERENCES file_registry(fingerprint)
);
```

**The hook chain:**

```
IDE Save / File Watcher Event
         │
         ▼
   chokidar 'change' event  (already exists in fileWatcher.js)
         │
         ▼
   FileWatcher._onFileChange()  (already fires on add/change/unlink)
         │
         ▼
   NEW: _emitMutation(change)  ← the hook point
         │
         ├──► persistence.logMutation()        → writes to file_mutation_log
         ├──► persistence.upsertFile()         → updates file_registry
         ├──► indexer.parseFile()              → re-extracts AST metadata
         ├──► schemaCardEmitter.emit(file)     → writes .st8/schema-card.json (deterministic, always in sync)
         └──► notificationBus.publish(event)   → SSE to frontend / notification
```

This is NOT a git hook. Git hooks only fire on `git commit` — they miss unsaved changes, mid-edit states, and the pre-code concept phase. The **file watcher** is the right hook point because it fires on every save, regardless of git state. It already exists in your codebase at [fileWatcher.js](/home/bozertron/1_AT_A_TIME/st8/backend/fileWatcher.js).

For the git integration specifically, we'd add a **complementary** `post-commit` hook that records commit-level state changes (like a checkpoint marker in the mutation log), but the file watcher is the primary driver.

**Why `.st8/schema-card.json` instead of `.md`:**
- Machine-readable — the indexer can read it back without parsing markdown
- Deterministic — always generated from source, never hand-edited, never drifts
- Diffs cleanly in git — you can `git diff` a schema card and see exactly what changed
- The `.st8/` directory is gitignored for production, or committed for development (your choice)

### 3. The Normalization Question: Fix the Code, Don't Paper Over It

You're absolutely right. Adding a conversion layer in `getAllFiles()` is a band-aid that adds complexity while the underlying inconsistency remains. Here's the correct approach:

**Phase 1: Define the canonical type** (`st8-types.js`)

```javascript
// st8-types.js — THE single source of truth
const St8FileEntry = {
  fingerprint:   '',       // stable identity (filepath:birth)
  filepath:      '',       // relative path
  filename:      '',       // basename
  sha256Hash:    '',       // content version
  fileSizeBytes: 0,
  status:        'RED',    // GREEN | YELLOW | RED | CONCEPT
  reachabilityScore: 0.0,
  impactRadius:  0,
  lastModified:  '',
  lastIndexed:   '',
  // Phase 2 additions:
  exports: [],             // from astParser
  imports: [],             // from astParser
  intent: null,            // from file_intent table
  mutationCount: 0,        // from file_mutation_log
};
```

**Phase 2: Fix the codebase directly**

| File | Current Problem | Fix |
|------|----------------|-----|
| `persistence.js` schema | `sha256_hash`, `file_size_bytes`, `reachability_score` | Rename columns to `sha256Hash`, `fileSizeBytes`, `reachabilityScore` |
| `persistence.js` `getAllFiles()` | Returns snake_case rows | Returns objects matching `St8FileEntry` shape |
| `persistence.js` `upsertFile()` | Mixes camelCase input with snake_case SQL | All camelCase, SQL uses aliases |
| `server.js` `_handleVerify` | `file.sha256_hash`, `file.file_size_bytes` | Use camelCase fields |
| `indexer.js` ST8_SCHEMA | snake_case column names | Align with `st8-types.js` |
| `index.js` change handler | Only updates `sha256Hash` | Updates all changed fields per `St8FileEntry` |

This is a **migration**: write a one-shot script that renames the SQLite columns and updates all SQL statements. The DB itself gets rebuilt on next index. For a codebase this size, it's a single session's work — no abstraction layer needed.

**Phase 3: Validate against `st8-types.js`**

Add a `validateSt8FileEntry(obj)` function that checks every field against the canonical shape. Run it in:
- `upsertFile()` — reject malformed data before it enters the DB
- `emitSchemaCard()` — guarantee the card matches the type
- A CLI command: `node st8-types.js --validate` — scan all DB rows and flag deviations

### 4. The Schema Card Emitter — Full AST Depth

The AST parser at [astParser.js](/home/bozertron/1_AT_A_TIME/st8/lib/utils/astParser.js) already extracts:

| Dimension | Data Available Now |
|-----------|-------------------|
| **Imports** | source, specifiers (name, alias, isType), importType (named/default/namespace/side-effect/dynamic/require) |
| **Exports** | name, kind (function/class/const/variable/type/interface/enum/default), signature, returnType, paramCount, typeParams |
| **Functions** | signature string, paramTypes, returnType, isPure (heuristic), complexity (cyclomatic + cognitive + LOC), JSDoc tags |
| **Arrow functions** | Same as functions |
| **Re-export chains** | Traces back to origin file, up to 5 levels deep |
| **Export star** | Resolves `export * from` up to 3 levels deep |
| **Dynamic imports** | `import()` calls detected via AST + regex fallback |
| **Vue SFC** | Extracts both `<script setup>` and `<script>` blocks |

That's **method-level granularity** already available. The schema card emitter just needs to call `extractImportsAndExports()` for each file and serialize the result.
